نسخهٔ **v2.2.0** — یک minor release که مهم‌ترین بهبودش پایداری مسیر apps_script در طول جلسه‌های طولانی است: heartbeat پس‌زمینه IP خروجی فیلتر‌شده را تشخیص و خودش جایگزین می‌کند، و pool مجوز tunnel-client حالا برای long-poll‌های idle و ترافیک فعال slot جدا نگه می‌دارد. علاوه‌بر این، decode اختیاری brotli/zstd، سقف ۳۲ MiB روی body درخواست، اعتبارسنجی پاسخ‌های DNS از تونل، و یک walkthrough جدید برای راه‌اندازی Full Tunnel هم اضافه شده. auto-updater این نسخه را روی نصب‌های موجود v2.1.x به‌صورت خودکار پیشنهاد می‌دهد.

• **مانیتور heartbeat پس‌زمینه برای سلامت IP خروجی Google با swap خودکار** ([322acbc](https://github.com/dazzling-no-more/rahgozar/commit/322acbc)). رله حالا هر ۳۰ ثانیه (پیش‌فرض `heartbeat_interval_secs`) یک probe سبک TCP+TLS+HEAD به `google_ip:443` می‌فرستد. اگر ۳ probe متوالی fail شوند (پیش‌فرض `heartbeat_failure_threshold`)، همان `scan_ips` که در دستور `scan-ips` اجرا می‌شود به‌صورت پس‌زمینه trigger می‌شود، اولین کاندیدای reachable که با SNI‌های `sni_hosts` سازگار باشد انتخاب می‌گردد، `google_ip` در حافظه swap می‌شود و connection pool + h2 cache پاک می‌شوند تا open‌های بعدی به IP جدید بروند. درخواست‌های در حال انجام روی IP قدیمی به‌طور طبیعی drain می‌شوند، و `config.json` روی دیسک دست‌نخورده باقی می‌ماند. پیش از این، اگر ISP در میانهٔ یک جلسه دامنهٔ datacenter جاری را filter می‌کرد، همه‌ی open‌های جدید fail می‌شدند تا کاربر برنامه را restart و `scan-ips` را دستی اجرا کند. سه کلید پیکربندی جدید (`heartbeat_enabled`، `heartbeat_interval_secs`، `heartbeat_failure_threshold`) برای tune کردن دقیق در شبکه‌های مختلف افزوده شده‌اند.

• **جداسازی سهمیهٔ فعال از idle در tunnel-client، عقب‌نشینی روی long-poll بی‌کار، و حذف close تکراری روی EOF** ([1c86df5](https://github.com/dazzling-no-more/rahgozar/commit/1c86df5)). یک حساب Apps Script کلاً ۳۰ execution همزمان به ازای deployment دارد، و در نسخه‌های قبلی long-poll‌های idle می‌توانستند همهٔ این slot‌ها را در زمان اوج ترافیک اشغال کنند و باعث queue شدن connect/upload‌های واقعی شوند. حالا یک کلاس مجوز دوم برای batch‌های idle جدا شده (پیش‌فرض ۲۴ slot برای idle، ۶ slot رزرو برای ترافیک فعال) و idle‌ها به‌جای صف اجبار، روی pool کل به‌صورت دوره‌ای retry می‌کنند. به‌علاوه، بعد از ۴ پاسخ خالی متوالی، تأخیر refill long-poll از صفر شروع به افزایش می‌کند و تا حداکثر ۷ ثانیه ramp می‌شود، که نرخ poll روی socket‌های واقعاً غیرفعال را بیش از نصف کاهش می‌دهد بدون اینکه socket‌های push-only کاملاً مرده به نظر برسند. به‌علاوه close ops که با EOF طبیعی session هم‌زمان می‌شوند redundant تلقی و حذف می‌گردند، که یک fetch اضافی در پایان هر اتصال صرفه‌جویی می‌کند.

• **رمزگشایی اختیاری brotli و zstd روی پاسخ‌های Apps Script** ([322acbc](https://github.com/dazzling-no-more/rahgozar/commit/322acbc)). پیش‌فرض همچنان حذف `br` و `zstd` از `Accept-Encoding` outbound است، چون `UrlFetchApp` فقط gzip را auto-decompress می‌کند و سایر encoding‌ها به‌صورت raw به مرورگر می‌رسند و خراب نمایش داده می‌شوند. این نسخه رمزگشاهای brotli و zstd داخل خود رله را اضافه می‌کند، و با set کردن `allow_brotli_zstd: true` در config، رله اجازه می‌دهد این encoding‌ها در `Accept-Encoding` forward شده باقی بمانند، body را server-side decode می‌کند و `Content-Encoding` را فقط در صورت موفقیت decode strip می‌نماید. مفید برای سایت‌هایی که CDN آن‌ها brotli را به gzip ترجیح می‌دهد (تا ~۲۰٪ payload کمتر روی leg مقصد). برای محافظت در برابر compression-bomb، خروجی decode‌شده به سقف ۶۴ MiB محدود است.

• **سقف ۳۲ MiB روی body درخواست HTTP/MITM با پاسخ ۴۱۳** ([d91cd74](https://github.com/dazzling-no-more/rahgozar/commit/d91cd74)). مسیرهای HTTP و MITM relay باید کل body درخواست را قبل از تحویل به Apps Script materialize کنند، و limit پایلود خود Apps Script هم بسیار پایین‌تر از "نامحدود" است وقتی overhead base64/JSON اضافه شود. این نسخه قبل از allocate کردن، `Content-Length` و طول cumulative chunked را با سقف ۳۲ MiB چک می‌کند و در صورت تجاوز، `413 Payload Too Large` برمی‌گرداند تا یک client محلی/LAN مخرب نتواند با درخواست‌های حجیم رله را وادار به allocate‌های عظیم کند. این سقف برای آپلودهای form/API معمولی به‌قدر کافی بالاست.

• **اعتبارسنجی پاسخ‌های DNS forward شده از تونل و سقف budget روی DoH در edge cache** ([2dd27eb](https://github.com/dazzling-no-more/rahgozar/commit/2dd27eb)). edge DNS cache روی Apps Script (که از v2.0.0 cache hit‌ها را قبل از round-trip به tunnel-node پاسخ می‌داد) حالا با سه بهبود ایمن‌تر و quota-friendly‌تر می‌شود: (۱) پاسخ‌های DNS که از tunnel-node برمی‌گردند قبل از این‌که در cache نوشته شوند parse و اعتبارسنجی می‌شوند، بنابراین یک reply مخرب/malformed نمی‌تواند cache را poison کند؛ (۲) cache hit‌های آن‌سو از tunnel-node حالا opportunistically cache می‌شوند تا batch بعدی همان name را local hit کند؛ (۳) یک budget سخت روی تعداد DoH fetch به ازای batch تنظیم می‌شود (پیش‌فرض ۱) تا یک burst سرد از name‌های مختلف یک fetch تونل را به N fetch DoH منفجر نکند. کلیدهای cache هم برای جلوگیری از lookup تکراری dedupe می‌شوند.

• **راهنمای جامع راه‌اندازی Full Tunnel از صفر** ([608706a](https://github.com/dazzling-no-more/rahgozar/commit/608706a)). فایل جدید `docs/full-tunnel-setup.md` (و نسخهٔ فارسی `full-tunnel-setup.fa.md`) تنها سندی است که برای راه‌اندازی Full Tunnel mode از صفر لازم دارید: VPS، Docker، deployment Apps Script CodeFull.gs، پیکربندی client، و آزمون end-to-end در حدود ۱۵ تا ۲۵ دقیقه. ارجاع‌های قدیمی به `ghcr.io/therealaleph/...` و repository upstream در سایر مستندات (راهنمای android، architecture، release-workflow، README پوشهٔ releases و tunnel-node) هم به مسیر فعلی `ghcr.io/dazzling-no-more/...` به‌روز شدند تا کاربری که از مستند قدیمی pull می‌کند تصویر اشتباه دریافت نکند.

**v2.2.0** is a minor release whose biggest improvement is `apps_script` path stability across long sessions: a background heartbeat detects when the egress IP gets newly filtered and swaps it in-place, and the tunnel-client permit pool now keeps separate slots for idle long-polls vs. active traffic. On top of that: opt-in brotli/zstd decode, a 32 MiB request body cap, validation of tunnel-forwarded DNS replies, and a new Full Tunnel setup walkthrough. The auto-updater will offer this to existing v2.1.x installs automatically.

• **Background heartbeat IP-health monitor with automatic swap** ([322acbc](https://github.com/dazzling-no-more/rahgozar/commit/322acbc)). The relay now sends one lightweight TCP+TLS+HEAD probe to `google_ip:443` every 30 s (default `heartbeat_interval_secs`). After 3 consecutive failures (default `heartbeat_failure_threshold`), it triggers the same `scan_ips` pass the `scan-ips` subcommand uses, picks the first reachable candidate that validates against any SNI in your `sni_hosts` pool, swaps `google_ip` in-memory, and clears the connection pool + h2 cache so subsequent opens target the new IP. In-flight requests on the old IP drain naturally; `config.json` on disk is left untouched. Previously, if an ISP newly filtered the current datacenter range mid-session, every new open would fail until the user restarted and re-ran `scan-ips` by hand. Three new config knobs (`heartbeat_enabled`, `heartbeat_interval_secs`, `heartbeat_failure_threshold`) let you tune detection latency vs. handshake cost per network.

• **Two-tier permit pool in tunnel-client, idle long-poll back-off, and skip-close on EOF** ([1c86df5](https://github.com/dazzling-no-more/rahgozar/commit/1c86df5)). A Google account gives you 30 concurrent Apps Script executions per deployment, and prior versions let idle long-polls occupy all of them during traffic spikes, queuing real connect/upload batches behind empty polls. There's now a second permit class for idle batches (default 24 idle slots, 6 reserved for active traffic), and idles retry the total pool periodically instead of queuing on Tokio's fair semaphore. After 4 consecutive empty responses a per-session idle refill delay ramps from 0 up to 7 s, cutting poll rate on truly quiet sockets by more than half without making push-only sockets feel dead. Close ops that coincide with natural session EOF are skipped — saving one final fetch per closing connection.

• **Opt-in brotli / zstd response decoding** ([322acbc](https://github.com/dazzling-no-more/rahgozar/commit/322acbc)). Default still strips `br` and `zstd` from outbound `Accept-Encoding` because `UrlFetchApp` only auto-decompresses gzip — other encodings would reach the browser as raw compressed bytes. This release ships in-relay brotli + zstd decoders; setting `allow_brotli_zstd: true` in config lets the relay forward those encodings, decodes the body server-side, and strips `Content-Encoding` only when decode succeeds. Helps on sites whose CDN prefers brotli over gzip (up to ~20% smaller payloads on the destination leg). Decoded output is capped at 64 MiB to defend against compression-bomb destinations.

• **32 MiB cap on HTTP/MITM request bodies with proper 413 response** ([d91cd74](https://github.com/dazzling-no-more/rahgozar/commit/d91cd74)). The HTTP and MITM relay paths have to materialize the full request body before handing it to Apps Script, and Apps Script's own payload limit is well below "unbounded" once base64/JSON overhead is included. The relay now checks `Content-Length` and cumulative chunked length against a 32 MiB cap before allocating, and replies `413 Payload Too Large` on excess — so a malicious local/LAN client cannot force huge allocations by sending oversized request bodies. The cap is generous enough for ordinary form/API uploads.

• **Tunnel-forwarded DNS replies are validated and DoH miss budget is bounded** ([2dd27eb](https://github.com/dazzling-no-more/rahgozar/commit/2dd27eb)). The Apps Script edge DNS cache (which has served cache hits without a round-trip to the tunnel-node since v2.0.0) gets three safety + quota improvements: (1) DNS replies coming back from the tunnel-node are parsed and validated before being written to the cache, so a malformed/spoofed reply cannot poison it; (2) those tunnel-forwarded hits are now also cached opportunistically so the next batch can serve the same name locally; (3) DoH fetches on cache miss are capped per batch (default 1) so a cold burst of distinct names cannot explode one tunnel fetch into N DoH fetches. Cache-key lookups within a batch are also deduplicated.

• **Full Tunnel setup walkthrough from zero** ([608706a](https://github.com/dazzling-no-more/rahgozar/commit/608706a)). New `docs/full-tunnel-setup.md` (and Persian `full-tunnel-setup.fa.md`) is the single document you need to set up Full Tunnel mode from scratch: VPS, Docker, CodeFull.gs Apps Script deployment, client config, and end-to-end test in ~15–25 minutes. Stale references to `ghcr.io/therealaleph/...` and the upstream repository in the rest of the docs (android guide, architecture, release-workflow, releases-folder README, tunnel-node README) are also corrected to the current `ghcr.io/dazzling-no-more/...` path so anyone following an older doc doesn't pull the wrong image.
