نسخهٔ **v2.0.2** — یک patch release با یک بهبود معماری بزرگ در tunnel client، چند fix در exit-node، و یک قابلیت تشخیص خودکار برای block شدن traffic توسط Cloudflare/Akamai. هیچ تغییر breaking ندارد و auto-updater آن را روی نصب‌های موجود v2.0.1 به‌صورت خودکار پیشنهاد می‌دهد.

• **پایپ‌لاین کردن poll‌های tunnel-node با depth انطباقی + ordering مبتنی بر `wseq` + block کردن STUN** ([8480ca5](https://github.com/dazzling-no-more/rahgozar/commit/8480ca5)). در Full mode، tunnel client حالا چند poll هم‌زمان به tunnel-node می‌فرستد و depth pipeline را بسته به شرایط شبکه به‌صورت زنده تنظیم می‌کند — به‌جای الگوی قدیم سریال «request → wait → response → next request» که در lossy linkها latency end-to-end را چندبرابر می‌کرد. هر write یک sequence number (`wseq`) می‌گیرد تا response‌ها در سمت client با ترتیب درست reassemble شوند، حتی وقتی response‌ها در شبکه out-of-order می‌رسند. به‌علاوه یک toggle جدید `block_stun` در config + UI اضافه شده (default در رهگذر `false` برای حفظ سازگاری) که اگر روشن کنید پورت‌های UDP 3478/5349/19302 در SOCKS5 listener block می‌شوند تا اپ‌های WebRTC مثل Google Meet، Discord و WhatsApp بدون تأخیر ۱۰-۳۰ ثانیه‌ای ICE timeout به TCP TURN روی پورت 443 fallback کنند. اندروید یک Pipeline Debug Overlay گرفته که در حالت debug عمق pipeline زنده را نمایش می‌دهد.

• **پشتیبانی از flag افزایشی `raw: true` در Apps Script و Cloudflare Worker برای مسیر بیرونی exit-node** ([cd78600](https://github.com/dazzling-no-more/rahgozar/commit/cd78600)). وقتی exit-node فعال است (چه در حالت `full` چه در حالت `selective` با لیست hosts)، رله یک hop بیرونی به Apps Script/Worker می‌زند که خود مقصدش یک exit-node است که body خود را در یک envelope `{s, h, b}` می‌پیچد — تا قبل از این، Apps Script آن envelope را یک بار دیگر می‌پیچید و client به‌جای محتوای صفحه، JSON خام دریافت می‌کرد. حالا با `raw: true`، envelope بیرونی unwrap می‌شود. این flag کاملاً additive است: کاربرانی که هیچ‌گاه exit-node فعال نمی‌کنند هیچ تغییری احساس نمی‌کنند. اما اگر `exit_node.enabled: true` در config دارید (مهم نیست `full` یا `selective`)، باید deployment Apps Script/Worker خود را نیز با نسخهٔ v2.0.2+ از `assets/apps_script/Code.gs` یا `assets/cloudflare/worker.js` redeploy کنید؛ deployment قدیمی `raw` را ignore می‌کند و رفتار double-wrap قدیمی را حفظ می‌کند که نتیجه‌اش نمایش JSON خام در browser است.

• **تشخیص خودکار block شدن توسط Cloudflare/Akamai در relay path** ([dcb0957](https://github.com/dazzling-no-more/rahgozar/commit/dcb0957)). یکی از مشکلات تکرارشوندهٔ کاربران این بوده که سایتی روی CF/Akamai مستقیماً Apps Script outbound IP را block می‌کند (Turnstile، `Just a moment...`، Akamai "Access Denied")، و کاربر بدون این‌که بفهمد علت چیست، گزارش «این سایت کار نمی‌کنه» می‌داد. ماژول جدید `bot_block.rs` در path رله ۱۶ کیلوبایت اول هر response را اسکن می‌کند و در صورت تشخیص signature بلاک، یک‌بار به ازای هر host یک پیام WARN در log می‌گذارد که به کاربر می‌گوید: این host را به `exit_node.hosts` اضافه کن تا از طریق exit-node خودت route شود. به‌علاوه `unity.com` به default curated groupها اضافه شد چون مثال canonical این الگو بود.

• **اصلاح default export در `exit_node.ts` برای Deno Deploy** ([0797919](https://github.com/dazzling-no-more/rahgozar/commit/0797919)). فرم قدیمی `export default async function` در Deno Deploy درست pickup نمی‌شد و کاربران deployهای جدید را با خطای cold-start مواجه می‌دیدند. حالا فایل یک object با property `fetch` export می‌کند که هم سینتکس مورد انتظار Deno Deploy است و هم با runtimeهای دیگر سازگار می‌ماند؛ `wrapper.ts` هر دو شکل را detect می‌کند تا deployهای قبلی نشکنند.

**v2.0.2** is a patch release with one large tunnel-client architectural improvement, a few exit-node fixes, and a new automatic detection for traffic blocked by Cloudflare / Akamai. No breaking changes; the auto-updater will offer this to existing v2.0.1 installs automatically.

• **Pipelined tunnel-node polls with adaptive depth, `wseq`-based ordering, and opt-in STUN blocking** ([8480ca5](https://github.com/dazzling-no-more/rahgozar/commit/8480ca5)). In Full mode, the tunnel client now keeps several polls in flight against tunnel-node concurrently and adapts pipeline depth live based on observed conditions — instead of the old serial "request → wait → response → next request" pattern that multiplied end-to-end latency on lossy links. Every write gets a sequence number (`wseq`) so the client can reassemble responses in the correct order even when they arrive out-of-order over the wire. A new `block_stun` toggle is added to config + UI (defaulting to `false` in rahgozar to preserve existing behavior); flipping it on blocks UDP ports 3478/5349/19302 at the SOCKS5 listener so WebRTC apps like Google Meet, Discord, and WhatsApp skip the 10–30 s ICE timeout and fall back to TCP TURN on 443 immediately. Android gained a Pipeline Debug Overlay that visualizes live pipeline depth in debug builds.

• **Additive `raw: true` flag on Apps Script + Cloudflare Worker for the exit-node outer hop** ([cd78600](https://github.com/dazzling-no-more/rahgozar/commit/cd78600)). When the exit-node is enabled (either `full` mode or `selective` mode with a `hosts` list), the relay's outer hop calls Apps Script / the Worker, whose destination is itself an exit-node that wraps its body in a `{s, h, b}` envelope — previously Apps Script would re-wrap that envelope, leaving the client with raw JSON instead of page content. With `raw: true` the outer envelope is unwrapped. The flag is fully additive: users who never enable the exit-node notice nothing. But if you have `exit_node.enabled: true` in your config (full *or* selective), you also need to redeploy your Apps Script / Worker from the v2.0.2+ [`assets/apps_script/Code.gs`](../../assets/apps_script/Code.gs) or [`assets/cloudflare/worker.js`](../../assets/cloudflare/worker.js) — older deployments ignore `raw` and keep the double-wrap behaviour, which surfaces as raw JSON in the browser. The client now detects this misconfig and surfaces a specific error pointing at the redeploy fix.

• **Automatic Cloudflare / Akamai bot-block detection on the relay path** ([dcb0957](https://github.com/dazzling-no-more/rahgozar/commit/dcb0957)). One recurring user complaint has been sites that CF / Akamai block outright when traffic originates from Apps Script's outbound IP space (Turnstile, `Just a moment...`, Akamai "Access Denied"), with no hint as to why "the site doesn't load through rahgozar." The new `bot_block.rs` module scans the first 16 KB of every relayed response on the path, and when a known block signature is hit it surfaces a one-shot per-host WARN in the log telling the user exactly what to do: add this host to `exit_node.hosts` so it routes through your own exit-node. `unity.com` was added to the curated defaults as the canonical Akamai example.

• **Object-with-`fetch` default export in `exit_node.ts` for Deno Deploy** ([0797919](https://github.com/dazzling-no-more/rahgozar/commit/0797919)). The old `export default async function` shape wasn't picked up correctly by Deno Deploy, so users provisioning fresh deployments would hit a cold-start error. The file now exports `{ fetch: handler }` — the shape Deno Deploy expects — while staying compatible with other runtimes; `wrapper.ts` detects both shapes so existing deployments don't break.
