نسخهٔ **v2.3.0** — یک minor release که تمرکز اصلی‌اش کاهش تأخیر هر RTT روی مسیر apps_script است، به‌ویژه برای handshakeهای چند-RTT (TLS، PSN sign-in، OAuth redirect). tunnel-node حالا DNS و TCP idle را برای مقاصد پُرتکرار pool می‌کند، rust client وقتی یک TLS handshake op در حال انجام است پنجرهٔ coalesce را تا ۵۰ms تنگ می‌کند، و selector deploymentهای Apps Script حالا با EWMA latency، deploymentهای کند را به‌طور موقت کنار می‌گذارد. علاوه‌بر این، یک flag ابزار اندازه‌گیری در CodeFull.gs برای بررسی floor تأخیر سمت Apps Script، و انتقال build داکر arm64 به runner بومی در CI (بدون QEMU) اضافه شده‌اند. auto-updater این نسخه را روی نصب‌های موجود v2.2.x به‌صورت خودکار پیشنهاد می‌دهد.

• **کش DNS و pool اتصال‌های idle TCP در tunnel-node برای مقاصد پُرتکرار** ([f2bfb36](https://github.com/dazzling-no-more/rahgozar/commit/f2bfb36)). پروفایلینگ یک session-open معمولی نشان داد که دو هزینهٔ مشخص روی هر handshake انباشته می‌شوند: حل DNS inline در `TcpStream::connect` (۵ تا ۳۰ms per resolve غیرکش‌شده) و سه‌مرحله‌ای TCP به یک edge دور CDN (۳۰ تا ۱۰۰ms). این نسخه یک `PrewarmState` به tunnel-node اضافه می‌کند: یک کش DNS با TTL ۶۰ ثانیه (سقف ۱۰۲۴ entry برای محافظت در برابر OOM) و یک pool اتصال‌های idle TCP با TTL ۳۰ ثانیه (تا ۲ اتصال در هر host، سقف کلی ۶۴) که فقط برای hostهای «hot» (≥۲ connect در پنجرهٔ ۵ دقیقه) فعال می‌شود تا FD budget روی tunnel-nodeهای اشتراکی محدود بماند. هر entry قبل از تحویل به session با یک non-blocking `try_read` چک می‌شود تا اتصال‌هایی که intermediary در میانه ghost کرده discard شوند. اگر یک پروتکل خاص با socketهای speculative بدرفتاری کند، با `MHRV_DISABLE_PREWARM=1` می‌شود کل لایه را غیرفعال کرد و به مسیر `connect_fresh` بازگشت.

• **تنگ کردن پنجرهٔ coalesce به ۵۰ms وقتی یک TLS handshake op در batch جاری است** ([a4338d4](https://github.com/dazzling-no-more/rahgozar/commit/a4338d4)). loop coalesce پیش‌فرض (سقف ۱ ثانیه) برای ترافیک bulk معامله‌ای منطقی بین quota و latency است، اما یک handshake TLS با ۲ تا ۳ RTT این coalesce را روی هر round-trip stack می‌کند و کل handshake از سمت برنامه ۳ تا ۶ ثانیه دیده می‌شود. اندازه‌گیری روی PSN sign-in از تونل: `time_appconnect` حدود ۶.۶ ثانیه در مقابل ~۰.۳ ثانیه مستقیم — تقریباً تمام این gap از stack coalesce می‌آمد. حالا opها به‌عنوان handshake-priority طبقه‌بندی می‌شوند: `Connect`، `ConnectData`، یا `Data` که byteهای ابتدایی‌اش `[0x16, 0x03, ..]` باشد (نوع record TLS Handshake + major version 3). وقتی هر op در batch priority باشد، deadline سخت batch به `min(coalesce_max, 50ms)` clamp می‌شود. clamp چسبنده و فقط shrink است — لنگرشده به instant اولین op، بنابراین یک priority op دیرآمده نمی‌تواند پنجره را extend کند، و یک `coalesce_max_ms` تنگ‌تر از طرف اپراتور هم محترم شمرده می‌شود. ترافیک bulk دست‌نخورده باقی می‌ماند: فقط خود handshake clamp را تریگر می‌کند.

• **کنار گذاشتن موقت deploymentهای Apps Script کند با امتیازدهی EWMA latency و انقضای نمونه** ([90d7865](https://github.com/dazzling-no-more/rahgozar/commit/90d7865)). فرونتر دامنه قبلاً همهٔ deploymentهای پیکربندی‌شده را به‌صورت round-robin استفاده می‌کرد، حتی اگر یکی از آن‌ها به‌خاطر throttle quota سمت Apps Script یا contention روی stream HTTP/2 برای یک دقیقه ۲ تا ۳ برابر کندتر شده باشد، که opهای handshake-stage را پشت‌سرهم stall می‌کرد. حالا یک EWMA per-deployment از RTT batchهای موفق (وزن آلفا ۰.۳) نگه داشته می‌شود، و یک slow-set از deploymentهایی که EWMA آن‌ها از median گروه > ۲× است، در `next_script_id` و `next_script_batch` skip می‌شوند. منطق skip تحت چند guard inert می‌ماند: کمتر از ۳ deployment پیکربندی‌شده (median معنی‌دار نیست)، median زیر ۵۰۰ms (همه سریعند، outlier هم batch قابل قبولی است)، یا وقتی هیچ deployment غیرکند/غیر-blacklist باقی نمانده (fallback به مسیر قبلی). هر نمونه فقط ۳۰ ثانیه «تازه» محسوب می‌شود، بنابراین یک deployment که برای یک دقیقهٔ بد skip شده، در سیکل بعد یک probe pick می‌گیرد و یا recover می‌کند یا برای ۳۰ ثانیهٔ دیگر بیرون می‌ماند — جلوی blacklist دائم به‌خاطر EWMA frozen گرفته می‌شود.

• **flag اختیاری `ENABLE_PERF_LOGGING` در CodeFull.gs برای ابزار اندازه‌گیری تأخیر سمت Apps Script** ([6432b45](https://github.com/dazzling-no-more/rahgozar/commit/6432b45)). هدف این flag پاسخ دادن به یک سؤال مشخص است: وقتی یک handshake کند است، هزینه در JS این فایل است (parse، dispatch، edge-DNS bookkeeping، serialize) یا در overhead ثابت Apps Script / `UrlFetchApp` که از این فایل قابل‌اصلاح نیست؟ با `ENABLE_PERF_LOGGING = true`، `doPost` کل wall-clock handler و مجموع مدت `UrlFetchApp.fetch` / `fetchAll` را اندازه می‌گیرد و یک خط grep-friendly به Stackdriver می‌فرستد: `perf: kind=tunnel_batch total=712 fetch=620 fetches=1 js=92 ops=5`. `js = total - fetch` هزینه‌ای است که ما مالکش هستیم و قابل بهینه‌سازی است؛ `fetch` بالا با `js` پایین یعنی به floor شبکه/Apps Script رسیده‌ایم. نام opهای tunnel در log از یک allowlist می‌آیند تا یک `req.t` خراب نتواند کاراکترهای کنترلی یا رشتهٔ بزرگ به Cloud Logging تزریق کند. پیش‌فرض `false` است، چون `console.log` تحت بار بالا overhead جزئی اما غیرصفر اضافه می‌کند — برای بررسی یک پنجرهٔ خاص روشن کنید، نمونه‌ها را جمع کنید، و دوباره خاموش کنید.

• **build docker arm64 به‌صورت بومی روی runner GitHub، حذف emulation QEMU** ([0ced312](https://github.com/dazzling-no-more/rahgozar/commit/0ced312)). job `tunnel-docker` در workflow release حالا به دو job split شده: `tunnel-docker-build` با matrix ۲-entry که هر platform را روی runner بومی خودش می‌سازد (`ubuntu-latest` برای amd64، `ubuntu-24.04-arm` برای arm64 — runner رایگان عمومی GitHub که از اوایل ۲۰۲۵ GA شده)، و `tunnel-docker-manifest` که digestهای per-platform را در یک manifest multi-arch واحد زیر tagهای release ترکیب می‌کند. نتیجه: build arm64 از ~۲۰ دقیقه تحت emulation QEMU به ~۳ تا ۵ دقیقهٔ بومی کاهش می‌یابد و مسیر critical tunnel-docker در release تقریباً نصف می‌شود. Dockerfile دست‌نخورده مانده — توسعه‌دهنده‌ای که محلی `docker build ./tunnel-node` یا `docker buildx build --platform=linux/amd64,linux/arm64` اجرا می‌کند هیچ تغییری حس نمی‌کند. ما فقط در CI emulate نمی‌کنیم.

**v2.3.0** is a minor release focused on cutting per-RTT latency on the apps_script path, especially for multi-RTT handshakes (TLS, PSN sign-in, OAuth redirects). tunnel-node now pools DNS + idle TCP for hot destinations, the rust client clamps the coalesce window to 50 ms whenever a TLS handshake op is in flight, and the Apps Script deployment selector now temporarily skips deployments running materially slower than peers via EWMA latency scoring. On top of that: an opt-in perf-logging flag in CodeFull.gs to measure the Apps Script floor, and the arm64 docker build is now native (no more QEMU) in CI. The auto-updater will offer this to existing v2.2.x installs automatically.

• **DNS cache + idle TCP pool in tunnel-node for hot destinations** ([f2bfb36](https://github.com/dazzling-no-more/rahgozar/commit/f2bfb36)). Profiling a typical session-open showed two real costs that compound on every handshake: inline DNS resolution in `TcpStream::connect` (5–30 ms per uncached resolve) and the TCP three-way handshake to a far CDN edge (30–100 ms). This release adds a `PrewarmState` to tunnel-node: a DNS cache with a 60 s TTL (capped at 1024 entries to defend against OOM) and an idle TCP pool with a 30 s TTL (up to 2 connections per host, global cap of 64) that only activates for "hot" hosts (≥ 2 connects in a 5-minute window) to bound FD usage on shared tunnel-nodes. Each pool entry is non-blocking `try_read`-checked before being handed to a session, so connections an intermediary has ghosted mid-idle are discarded. If a specific protocol misbehaves with speculative upstream sockets, `MHRV_DISABLE_PREWARM=1` turns the whole layer off and falls back to the `connect_fresh` path.

• **Coalesce window tightened to 50 ms while a TLS handshake op is in flight** ([a4338d4](https://github.com/dazzling-no-more/rahgozar/commit/a4338d4)). The default coalesce window (1 s max) is a reasonable quota/latency trade for bulk traffic, but a 2-3 RTT TLS handshake stacks that coalesce on every round-trip, pushing the application-visible handshake to 3–6 s. Measured on PSN sign-in through the tunnel: `time_appconnect` of ~6.6 s vs ~0.3 s direct — almost the entire gap was coalesce stacking. Ops are now classified as handshake-priority: `Connect`, `ConnectData`, or `Data` whose leading bytes are `[0x16, 0x03, ..]` (TLS record-layer Handshake content type + major version 3). When any op in a batch is priority, the batch's hard deadline is clamped to `min(coalesce_max, 50 ms)`. The clamp is sticky and shrink-only — anchored to the first-op instant, so a late priority op cannot extend the window, and a tighter operator-configured `coalesce_max_ms` is honored. Bulk traffic is untouched: only the handshake itself triggers the clamp.

• **Skip slow Apps Script deployments via EWMA latency scoring with sample expiry** ([90d7865](https://github.com/dazzling-no-more/rahgozar/commit/90d7865)). The domain fronter used to round-robin every configured deployment unconditionally, even when one of them had slowed down 2–3× for a minute due to Apps Script quota throttling or HTTP/2 stream contention — stalling handshake-stage ops behind it repeatedly. There's now a per-deployment EWMA of successful-batch RTT (alpha = 0.3), and deployments whose EWMA exceeds the group median by > 2× are added to a slow-set that `next_script_id` and `next_script_batch` skip. Skip logic is inert under several guards: fewer than 3 configured deployments (median isn't meaningful), median below 500 ms (everyone's already fast — an outlier is still a fine batch), or no non-slow/non-blacklisted deployments left (fall through to the prior path). Each sample is only "fresh" for 30 s, so a deployment skipped for a bad minute gets a probe pick on the next cycle and either recovers or stays out for another 30 s — preventing permanent blacklisting from a frozen high EWMA.

• **Opt-in `ENABLE_PERF_LOGGING` flag in CodeFull.gs for Apps-Script-side timing** ([6432b45](https://github.com/dazzling-no-more/rahgozar/commit/6432b45)). The flag exists to answer one specific question: when a handshake is slow, is the cost in this file's JS (parse, dispatch, edge-DNS bookkeeping, serialize) or in fixed Apps Script / `UrlFetchApp` overhead we can't fix from here? With `ENABLE_PERF_LOGGING = true`, `doPost` records the total handler wall-clock and the sum of `UrlFetchApp.fetch` / `fetchAll` durations, then emits one grep-friendly line to Stackdriver: `perf: kind=tunnel_batch total=712 fetch=620 fetches=1 js=92 ops=5`. `js = total - fetch` is the cost we own and can optimize; high `fetch` with low `js` means we've hit the network/Apps Script floor. Tunnel op names in the log come from an allowlist so a malformed `req.t` can't smuggle control characters or oversize strings into Cloud Logging. The default is `false` because `console.log` adds a tiny but non-zero overhead under heavy load — turn it on to investigate a specific window, capture samples, and turn it off again.

• **Build tunnel-docker arm64 natively on GitHub runners, drop QEMU emulation** ([0ced312](https://github.com/dazzling-no-more/rahgozar/commit/0ced312)). The release workflow's `tunnel-docker` job is split into two: `tunnel-docker-build` with a 2-entry matrix that builds each platform on its own native runner (`ubuntu-latest` for amd64, `ubuntu-24.04-arm` for arm64 — GitHub's free public arm64 runner, GA since early 2025), and `tunnel-docker-manifest` which stitches the per-platform digests into one multi-arch manifest under the release tags. Result: arm64 build drops from ~20 min under QEMU emulation to ~3–5 min native, cutting the tunnel-docker critical path of the release roughly in half. The Dockerfile is unchanged — a developer running `docker build ./tunnel-node` or `docker buildx build --platform=linux/amd64,linux/arm64` locally sees no difference. We just don't emulate in CI anymore.
