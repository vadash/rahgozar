نسخهٔ **v2.6.0** — یک minor release که حالت جدید `local_bypass` را معرفی می‌کند (دور زدن DPI به‌صورت کاملاً محلی، بدون نیاز به Apps Script یا گواهی MITM) و یک مشکل reliability را در مسیر domain fronting برطرف می‌کند که اتصال‌های h2 را پس از blackhole شدن TCP توسط middlebox ISP، بی‌سر و صدا خراب می‌گذاشت. هیچ تغییر breaking در API یا config وجود ندارد — نصب‌های فعلی CLI، دسکتاپ و اندروید در محل آپدیت می‌شوند.

• **حالت جدید `local_bypass` — دور زدن DPI بدون رله و بدون گواهی** ([7cc63fa](https://github.com/dazzling-no-more/rahgozar/commit/7cc63fa)). یک مود چهارم در کنار `apps_script`، `direct` و `full`. هر TLS handshake به‌صورت محلی تکه‌بندی می‌شود و مستقیماً به مقصد واقعی می‌رود — نه deploy نیاز است، نه auth key، نه نصب گواهی MITM، و `cert pinning` واقعی روی هاست‌های مقصد دست‌نخورده می‌ماند. در اپ اندروید، ترافیک همهٔ اپ‌ها به‌صورت خودکار از طریق VpnService گرفته می‌شود؛ در دسکتاپ، فقط اپ‌هایی که از system proxy (`127.0.0.1:8085`) استفاده می‌کنند سود می‌برند (مرورگرها و بیشتر اپ‌های آگاه به پروکسی). selectorهای حالت در UI اندروید و دسکتاپ، و یک [config.local_bypass.example.json](../../config.local_bypass.example.json) هم اضافه شده‌اند. تذکر مهم: فقط DPI را دور می‌زند، نه انسداد سطح IP — پس مقصدهایی مثل `claude.ai` / `x.ai` / `chatgpt.com` که در ایران در سطح IP بسته‌اند با این حالت باز نمی‌شوند و همچنان به `apps_script` یا `full` نیاز دارند.

• **تشخیص اتصال‌های h2 «بی‌صدا مرده» در domain fronting با PING liveness** ([08e6ba6](https://github.com/dazzling-no-more/rahgozar/commit/08e6ba6)). روی ISPهای ایران رایج است که middlebox، RSTهای یک اتصال طولانی به YouTube / x.com / Telegram-web را بخورد. در این حالت کانکشن h2 از دید کد «زنده» باقی می‌ماند و هر درخواست در صف منتظر یک `read` بی‌پاسخ می‌مانَد تا برنامه ری‌استارت شود. حالا یک PING در سطح اپلیکیشن هر ۱۵ ثانیه فرستاده می‌شود و در صورت نرسیدن PONG ظرف ۱۰ ثانیه، فلگ `dead` ست می‌شود تا `ensure_h2` در همان tick بعدی refill اتصال را تعویض کند. بیشترین تأخیر تشخیص یک سوکت کاملاً مرده حدود ۲۵ ثانیه است (interval + timeout)، در حالی که RTT تیپیکال ایران→گوگل بین ۲۰۰ تا ۴۰۰ میلی‌ثانیه است — یعنی jitter موبایل/WiFi باعث false-positive close نمی‌شود.

**v2.6.0** is a minor release that introduces a new `local_bypass` mode (fully-local DPI bypass with no Apps Script and no MITM cert) and fixes a reliability bug in the domain-fronting path where h2 connections could be left silently broken after an ISP middlebox blackholed the underlying TCP socket. No breaking API or config changes — existing CLI, desktop, and Android installs upgrade in place.

• **New `local_bypass` mode — DPI bypass without a relay and without a cert** ([7cc63fa](https://github.com/dazzling-no-more/rahgozar/commit/7cc63fa)). A fourth mode alongside `apps_script`, `direct`, and `full`. Every TLS handshake gets fragmented locally and sent direct to the real destination — no deploy required, no auth key, no MITM cert to install, and real `cert pinning` on destination hosts keeps working untouched. On the Android app, every app's traffic is captured automatically via VpnService; on desktop, only apps that honor the system proxy (`127.0.0.1:8085`) benefit (browsers and most proxy-aware apps). Mode selectors land in both Android and desktop UIs, plus a [config.local_bypass.example.json](../../config.local_bypass.example.json) for CLI users. Important caveat: it only beats DPI, not IP-level blocks — destinations like `claude.ai` / `x.ai` / `chatgpt.com` that Iran blocks at the IP layer won't open in this mode and still need `apps_script` or `full`.

• **Detect silently-dead h2 connections in domain fronting via PING liveness** ([08e6ba6](https://github.com/dazzling-no-more/rahgozar/commit/08e6ba6)). On Iranian ISPs it's common for a middlebox to eat the RSTs of a long-lived h2 flow to YouTube / x.com / Telegram-web. The h2 connection looked "alive" to the code, and every queued request would block on a read that never errors until the app was restarted. An application-level PING is now sent every 15 s, and if no PONG arrives within 10 s the `dead` flag is set so `ensure_h2` swaps the connection on the next refill tick. Worst-case detection latency for a fully-dead socket is ~25 s (interval + timeout); typical Iran→Google RTT is 200–400 ms, so mobile/WiFi jitter won't false-positive-close a healthy connection.
