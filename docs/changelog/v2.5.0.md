• **کاهش latency راه‌اندازی full tunnel با یک optimistic pipeline سه‌slotه** ([4101fc4](https://github.com/dazzling-no-more/rahgozar/commit/4101fc4)، [10d5957](https://github.com/dazzling-no-more/rahgozar/commit/10d5957)). `connect_data` اتصال TCP و اولین byteهای کلاینت را در همان batch اول بسته‌بندی می‌کند؛ بعد از برگشت آن، poll اولیه و دو upload کوچک بعدی (مثل TLS Finished و request رمز‌شده) با هم در اولین batch پس از connect حرکت می‌کنند. برای HTTPS تازه، اولین byte پاسخ معمولاً همچنان cycle دوم Apps Script است، اما این مسیر cycle سوم اضافه را حذف می‌کند و startup مربوط به HTTP ساده را در همان درخواست `connect_data` جا می‌دهد.

• **صرفه‌جویی حدود ۳۰٪ در دانلود به‌کمک batchهای فشرده‌شده با zstd در full tunnel** ([78a0f22](https://github.com/dazzling-no-more/rahgozar/commit/78a0f22)، port از upstream MasterHttpRelayVPN#1314). پاسخ‌هایی که از طریق full tunnel دریافت می‌شوند بین Apps Script relay و کلاینت به‌صورت end-to-end با zstd فشرده می‌شوند. چون هر دو سمت `TUNNEL_AUTH_KEY` را در اختیار دارند، از crate سی-بک‌اند `zstd` به جای decoder pure-Rust استفاده شده — هزینهٔ CPU پایین‌تر و بدون نگرانی origin-controlled-input.

• **بهبود reliability مسیر tunnel** ([7db3d7a](https://github.com/dazzling-no-more/rahgozar/commit/7db3d7a)، [25ccbe6](https://github.com/dazzling-no-more/rahgozar/commit/25ccbe6)، [df71cc8](https://github.com/dazzling-no-more/rahgozar/commit/df71cc8)). خطاهای transient TLS handshake در مرحلهٔ startup مربوط به connect_data الان retry می‌شوند به‌جای اینکه به‌صورت یک اتصال failed بالا بیایند. deploymentهای موفقی که کند اجرا می‌شوند یک cool-down کوتاه trigger می‌کنند تا pipeline بیش از حد به relay فشار نیاورد. در طول uploadهای فعال دیگر empty poll barrier ارسال نمی‌شود — یک round-trip اضافی از cadence سمت upload حذف شده است.

**v2.5.0** is a minor release focused on full-tunnel speed and reliability: an optimistic 3-slot pipeline cuts startup latency, zstd batch compression saves ~30% of download traffic, and a handful of TLS / deployment reliability bugs are fixed. No breaking API or config changes — existing CLI, desktop, and Android installs upgrade in place.

• **Full-tunnel startup latency cut with an optimistic 3-slot pipeline** ([4101fc4](https://github.com/dazzling-no-more/rahgozar/commit/4101fc4), [10d5957](https://github.com/dazzling-no-more/rahgozar/commit/10d5957)). `connect_data` bundles TCP connect with the client's first bytes in the first batch; after it returns, the initial poll and two small follow-up uploads (for example TLS Finished plus the encrypted request) ride together in the first post-connect batch. Fresh HTTPS normally still needs the second Apps Script cycle for first response data, but this removes the extra third-cycle stall and still lets plain-HTTP startup fit into the `connect_data` request itself.

• **~30% download saving via zstd-compressed Full tunnel batches** ([78a0f22](https://github.com/dazzling-no-more/rahgozar/commit/78a0f22), port of upstream MasterHttpRelayVPN#1314). Reply bodies fetched through the full tunnel are zstd-compressed end-to-end between the Apps Script relay and the client. Since both sides hold `TUNNEL_AUTH_KEY`, this uses the C-backed `zstd` crate rather than the pure-Rust decoder — lower CPU cost and no origin-controlled-input concerns.

• **Tunnel reliability improvements** ([7db3d7a](https://github.com/dazzling-no-more/rahgozar/commit/7db3d7a), [25ccbe6](https://github.com/dazzling-no-more/rahgozar/commit/25ccbe6), [df71cc8](https://github.com/dazzling-no-more/rahgozar/commit/df71cc8)). Transient TLS handshake failures during the connect_data startup are now retried instead of bubbling up as a failed connection. Batch deployments that succeed but run slowly trigger a brief cool-down so the pipeline stops over-driving the relay. Empty poll barriers no longer fire during active uploads, removing a redundant round-trip from the upload-side cadence.
