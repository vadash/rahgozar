نسخهٔ **v2.10.0** — یک minor release که دسترسی **بدون رله و با سرعت کامل** به یوتیوب و متا را با روش «استتار» اضافه می‌کند (الهام‌گرفته از [@patterniha](https://github.com/patterniha/MITM-DomainFronting)).

• **مسیر «استتار» (force_ip) برای دامنه‌هایی که edge مشترک قابل‌fronting ندارند** ([ee10435](https://github.com/dazzling-no-more/rahgozar/commit/ee10435)). ویدیوی یوتیوب (`googlevideo.com` روی edge جدای «EVA») و اینستاگرام/واتس‌اپ/فیسبوک edge مشترکی ندارند که بشود IP ثابتی برایش پین کرد، برای همین تا الان یا از رله (کند) رد می‌شدند یا اصلاً باز نمی‌شدند. حالا این دامنه‌ها مستقیم به IP واقعی خودشان وصل می‌شوند، با یک SNI تقلبی (`www.google.com` / `www.microsoft.com`) که فقط چشم DPI را می‌بندد؛ گواهی واقعیِ همان میزبان راستی‌آزمایی می‌شود، پس IP عوض‌شده یا مسموم نمی‌تواند وسط راه بنشیند.

• **resolver داخلی DoH (استتارشده) برای پیدا کردن IP واقعی بدون مسمومیت DNS.** چون DNS سیستم در ایران دقیقاً برای همین میزبان‌ها مسموم است، یک resolver کوچک DoH اضافه شد که خودِ اتصالش هم استتارشده است و سه نقطه را موازی امتحان می‌کند: Cloudflare `1.1.1.1` و `1.0.0.1` (استتارشده) به‌علاوهٔ Google `dns.google`. اگر DoH یا خودِ IP مقصد مسدود باشد، اتصال بی‌صدا به مسیر عادی (رله در حالت apps_script، یا TCP خام در حالت direct) برمی‌گردد و قطع نمی‌شود؛ یک breaker برای هر گروه هم جلوی گیر کردن روی شبکه‌ای که آن IPها را به‌کلی drop می‌کند را می‌گیرد.

• **گروه‌های آمادهٔ `google-video` و `meta` به بستهٔ fronting اضافه شدند** — با دکمهٔ «بارگذاری گروه‌های fronting آماده» (دسکتاپ و اندروید) فعال می‌شوند. این مسیر فقط مسدودسازی مبتنی‌بر SNI را دور می‌زند؛ اگر ISP خودِ IP را مسدود کند، رله همچنان تنها گزینه می‌ماند.

**v2.10.0** is a minor release that adds **relay-free, full-speed** access to YouTube and Meta via a "camouflage" route (inspired by [@patterniha](https://github.com/patterniha/MITM-DomainFronting)).

• **"Camouflage" (force_ip) route for domains with no frontable shared edge** ([ee10435](https://github.com/dazzling-no-more/rahgozar/commit/ee10435)). YouTube video (`googlevideo.com`, on the separate "EVA" edge) and Instagram/WhatsApp/Facebook have no shared edge IP you can pin, so until now they either went through the relay (slow) or didn't open at all. They now dial their **real** IP directly with a **decoy SNI** (`www.google.com` / `www.microsoft.com`) that only blinds the DPI box; the real host's certificate is still verified, so a wrong/poisoned IP fails closed instead of MITMing you.

• **Built-in camouflaged DoH resolver to find the real IP without DNS poisoning.** Because the system resolver in Iran is poisoned for exactly these hosts, a small DoH client was added — its own handshake camouflaged too — racing three endpoints in parallel: Cloudflare `1.1.1.1` and `1.0.0.1` (camouflaged) plus Google `dns.google`. If DoH or the destination IP is blocked, the connection silently **falls through** to the normal route (relay in apps_script, raw TCP in direct) rather than dropping; a per-group breaker also avoids stalling on a network that blackholes those IPs.

• **New `google-video` and `meta` curated fronting groups** — enabled via "Load curated fronting groups" (desktop and Android). This route defeats **SNI-based** blocking only; if an ISP IP-blocks the destination outright, the relay remains the fallback.
