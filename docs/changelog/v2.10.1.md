نسخهٔ **v2.10.1** — رفعِ «اسکرول یوتیوب فقط می‌چرخد و چیزی لود نمی‌شود». فیدِ وب‌اپِ یوتیوب به HTTP/2 نیاز دارد و روی مسیر قبلی (که قفلِ HTTP/1.1 بود) گیر می‌کرد.

• **HTTP/2 روی مسیر استتار، با هماهنگیِ ALPN** ([#16](https://github.com/dazzling-no-more/rahgozar/pull/16)). تونلِ استتار حالا ClientHelloی مرورگر را peek می‌کند (بدون مصرف، تا fall-through حفظ شود)، به upstream فقط همان پروتکل‌هایی را پیشنهاد می‌دهد که مرورگر پیشنهاد داده، و در نهایت دقیقاً همان پروتکلی را که edge انتخاب کرده به مرورگر می‌دهد — پس هر دو سرِ splice همیشه روی یک پروتکل توافق دارند. اگر peek ناقص بود، هر دو سر به http/1.1 تنزل می‌کنند (هیچ‌وقت ناهماهنگی).

• **گروهِ آمادهٔ `youtube-web`.** دامنه‌های `youtube.com`، `youtubei.googleapis.com`، `ytimg.com` و … حالا از مسیرِ استتار (ForceIP + HTTP/2) عبور می‌کنند، نه مسیرِ رله/SNI-rewrite که فقط h1 بود. همین فیدِ اسکرول را باز می‌کند. با دکمهٔ «بارگذاری گروه‌های fronting آماده» فعال می‌شود.

• **تغییر رفتارِ `verify_names` (ناسازگار با قبل):** قبلاً یک لیستِ غیرخالی، میزبانِ واقعی را *جایگزین* می‌کرد؛ حالا *به‌علاوهٔ* میزبانِ واقعی پذیرفته می‌شود. این فقط حالتِ سخت‌گیرانه‌تر را شل‌تر می‌کند (همچنان fail-closed؛ سانسورگر نمی‌تواند گواهیِ معتبرِ عمومی جعل کند). اگر `verify_names` را دستی نوشته بودی تا میزبان را عمداً حذف کنی، حالا میزبان هم پذیرفته می‌شود. گروه‌های آماده نامِ SNI تقلبی را اینجا pin می‌کنند (`www.google.com` / `www.microsoft.com`).

**v2.10.1** fixes "scrolling YouTube just spins, nothing loads." The YouTube web app's feed needs HTTP/2 and stalled on the previous path, which was locked to HTTP/1.1.

• **HTTP/2 across the camouflage splice, with ALPN coherence** ([#16](https://github.com/dazzling-no-more/rahgozar/pull/16)). The camouflage tunnel now peeks the browser's ClientHello (non-consuming, so fall-through is preserved), offers the upstream only the protocols the browser offered, and finally hands the browser exactly the protocol the edge chose — so both legs of the raw splice always agree. A partial peek degrades both legs to http/1.1 (never a mismatch).

• **New `youtube-web` curated group.** `youtube.com`, `youtubei.googleapis.com`, `ytimg.com`, etc. now go through the camouflage path (ForceIP + HTTP/2) instead of the relay/SNI-rewrite path (which is h1-only) — this is what makes the feed load. Enabled via "Load curated fronting groups".

• **`verify_names` behavior change (backward-incompatible):** a non-empty `verify_names` previously *replaced* the real destination host; it now applies *in addition to* it. This only loosens a stricter setting (still fail-closed — a censor can't forge a valid public cert). If you hand-wrote `verify_names` to deliberately exclude the host, the host is now accepted again. The curated groups use it to pin the decoy SNI name (`www.google.com` / `www.microsoft.com`).
