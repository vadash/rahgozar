نسخهٔ **v2.8.0** — یک minor release که Drive Mode را معرفی می‌کند: یک حالت تونل جدید که ترافیک TLS را از طریق یک پوشهٔ مشترک Google Drive منتقل می‌کند، برای ISPهایی که `googleapis.com` را اجازه می‌دهند ولی به IP اختصاصی VPS شما دسترسی نمی‌دهند. سه پاس بهبود کارایی پشت سر هم، latency cold TLS handshake را از ~۶.۳ ثانیه به ~۵.۱ ثانیه و warm TTFB را از ~۶.۱۵ ثانیه به ~۴.۳۵ ثانیه روی یک VPS معمول از ایران رسانده‌اند. هیچ تغییر breaking در حالت‌های موجود (Apps Script، Full، Direct) نیست — Drive Mode به‌صورت opt-in در کنار آن‌ها قرار می‌گیرد.

• **افزودن Drive Mode — حالت mailbox از طریق Google Drive** ([d6204fc](https://github.com/dazzling-no-more/rahgozar/commit/d6204fc)). یک حالت transport جدید با الهام از Skirk: کلاینت فریم‌های TLS را به‌صورت رمزشده (AEAD، X25519 برای session keys) به‌عنوان فایل‌های `c2r_<sid>_<seq>` در یک پوشهٔ Drive آپلود می‌کند؛ یک پروسهٔ `rahgozar-drive-relay` روی VPS شما پوشه را poll می‌کند، فریم‌ها را باز می‌کند، به مقصد واقعی dial می‌زند، و پاسخ را به‌صورت `r2c_*` آپلود می‌کند. مناسب برای کاربرانی که ISP آن‌ها لیست سفید سختی دارد ولی هنوز به Drive API دسترسی دارد. راهنمای راه‌اندازی در `docs/drive_oauth_setup.md`.

• **بهبود کارایی Drive Mode در سه پاس متوالی** ([de4e22e](https://github.com/dazzling-no-more/rahgozar/commit/de4e22e)). پاس اول ([ddd4e4a](https://github.com/dazzling-no-more/rahgozar/commit/ddd4e4a)) چند فریم را در یک آپلود Drive باندل می‌کند، query لیست‌گیری را به یک cursor مبتنی بر `modifiedTime` (~۱۰ برابر ارزان‌تر از مسیر full-text `name contains`) سوییچ می‌کند، و idle backoff را وقتی session فعال است حذف می‌کند. پاس دوم ([238c6e4](https://github.com/dazzling-no-more/rahgozar/commit/238c6e4)) پیام Hello باز کردن session را در بدنهٔ همان آپلود `c2r_0` می‌گنجاند (یک round-trip Drive کمتر در cold start) و TLS pool به `www.googleapis.com` را pre-warm می‌کند تا اولین آپلود hit handshake سرد نخورد. پاس سوم ([de4e22e](https://github.com/dazzling-no-more/rahgozar/commit/de4e22e)) سقف idle-poll را از ۱۵۰۰ms به ۵۰۰ms و pipeline interval relay را از ۱۰۰ms به ۲۵ms کاهش می‌دهد (مطابق سمت کلاینت). نتیجهٔ ترکیبی: کاهش ~۳۰٪ در warm TTFB.

• **ابزار release — بازطراحی body composer و حذف هدف MIPS شکستهٔ upstream** ([52acc74](https://github.com/dazzling-no-more/rahgozar/commit/52acc74)). صفحهٔ release جدید به‌سبک uv برای هر پلتفرم یک dropdown اختصاصی نشان می‌دهد به‌جای لیست تخت طولانی؛ فایل‌های `.minisig` در dropdown مخفی می‌شوند و فقط در زیر لینک باینری اصلی نمایش داده می‌شوند تا کاربر معمولی گیج نشود. هم‌چنین هدف `mips-unknown-linux-musl` از ماتریکس CLI ([a3f7fa4](https://github.com/dazzling-no-more/rahgozar/commit/a3f7fa4)) حذف شد چون toolchain upstream شکسته است و کاربر شناخته‌شده‌ای روی این arch ندارد؛ MIPSEL (little-endian، حالت soft-float، اضافه‌شده در v2.7.0) همچنان پشتیبانی می‌شود.

**v2.8.0** is a minor release that introduces Drive Mode: a new tunnel transport that ferries TLS traffic through a shared Google Drive folder, for ISPs that allow `googleapis.com` but block direct egress to your VPS IP. Three back-to-back performance passes bring the cold TLS handshake from ~6.3 s to ~5.1 s and warm TTFB from ~6.15 s to ~4.35 s on a typical Iran VPS. No breaking changes to existing modes (Apps Script, Full, Direct) — Drive Mode is opt-in alongside them.

• **Add Drive Mode — a Google Drive mailbox transport** ([d6204fc](https://github.com/dazzling-no-more/rahgozar/commit/d6204fc)). A Skirk-inspired transport: the client uploads AEAD-sealed (X25519-derived session keys) TLS frames as `c2r_<sid>_<seq>` files into a shared Drive folder; a `rahgozar-drive-relay` process running on your VPS polls the folder, opens the frames, dials the real destination, and posts the responses back as `r2c_*`. Aimed at users on ISPs whose whitelist is strict but still permits the Drive API. Setup guide in `docs/drive_oauth_setup.md`.

• **Improve Drive Mode latency across three passes** ([de4e22e](https://github.com/dazzling-no-more/rahgozar/commit/de4e22e)). Pass 1 ([ddd4e4a](https://github.com/dazzling-no-more/rahgozar/commit/ddd4e4a)) coalesces multiple frames into one Drive upload, switches the list query to a `modifiedTime` cursor (~10× cheaper per call than the full-text `name contains` path), and skips idle backoff while at least one session is active. Pass 2 ([238c6e4](https://github.com/dazzling-no-more/rahgozar/commit/238c6e4)) folds the session-open Hello prefix into the same `c2r_0` upload body — one fewer Drive round-trip on cold start — and pre-warms the TLS pool to `www.googleapis.com` so the very first upload doesn't pay a cold handshake. Pass 3 ([de4e22e](https://github.com/dazzling-no-more/rahgozar/commit/de4e22e)) drops the idle-poll cap from 1500 ms → 500 ms and the relay's active-session pipeline interval from 100 ms → 25 ms (matching the client side). Combined effect: ~30% warm-TTFB reduction.

• **Release tooling: redesigned body composer + dropped a broken MIPS target** ([52acc74](https://github.com/dazzling-no-more/rahgozar/commit/52acc74)). The GitHub release page now uses uv-style per-platform dropdowns instead of one long flat asset list; `.minisig` signature files are hidden from the dropdowns and surfaced only under their parent binary so the casual visitor isn't confused by twice as many entries. Separately, the `mips-unknown-linux-musl` target was dropped from the CLI matrix ([a3f7fa4](https://github.com/dazzling-no-more/rahgozar/commit/a3f7fa4)) because the upstream toolchain is broken and we have no known users on that arch; MIPSEL (little-endian, soft-float, added in v2.7.0) remains supported.
